Why This Training

Stratpoint needs organizational AI capability.
Not individual AI users.

Individual practitioners using AI well is not the same as an organization that delivers with AI. This training builds the second thing — a shared framework, a shared quality bar, and a knowledge base that compounds across every project and team.
  • Every role in the pipeline, not just developers. Solutions Designers, PMs, SAs, Tech Leads, and Developers all producing AI-assisted artifacts with quality gates they own — not watching from the sidelines.
  • Methodology over tools. Prompting discipline, context engineering, and the artifact governance framework transfer across any AI tool. What you learn here applies regardless of which tool you're handed on the next project.
  • Traceability from business requirement to deployed code. Every feature must be walkable from a BRD requirement through PRD, Architecture, Dev Tasks, code, and tests. A broken chain is a problem — not a style choice.
  • Skills that outlast the person who found them. Prompts, agents, skills, and rules committed to a shared knowledge base. The next team inherits the playbook — not a blank slate.
  • A knowledge base that compounds. Every cohort adds to what the last one built. The organization improves at AI-augmented delivery with every team that runs through this program.
  • First-wave evangelists who set the standard. This cohort doesn't just learn the framework — they define how Stratpoint uses AI on client engagements going forward. What you build and commit this week is the bar every future team starts from.

The Framework

One shared structure. AI generates. Humans decide. Everything traces.

Not a new tool. A new way of working with the tools you already have — structured, traceable, and built to compound across every project that uses it.
BRD→ PRD→ Architecture→ Dev Tasks→ Code→ Tests→ Deploy
What holds it together
  • Named human owners at every artifact — not "the team." The artifact's owner signs it off.
  • Quality gates at every phase transition — not deferred to the end. A gate fails → work goes back, not forward.
  • Full traceability — any line of code can be walked back to its BRD requirement. If it can't, the chain is broken and must be fixed.
What this training produces
  • A complete artifact chain — BRD to deployed code — produced under the framework on a real sample project
  • Every participant working in their role's domain with AI in a structured, repeatable way
  • A shared knowledge base — agents, skills, prompts, rules — that outlasts this cohort and compounds with every one that follows

PRINCIPLES
Non-negotiable · All phases · All roles
Four Non-Negotiable Principles

These govern every decision in the framework.

No exceptions. Not suggestions.
01
AI Augments, Humans Decide
Every AI output passes through human review before it moves downstream. AI never owns the decisions that carry formal accountability — the named artifact owner does.
02
Structure Before Speed
Invest in artifacts before engineering begins. Teams that skip upfront structure feel fast in Week 1 and stall in Week 3. The BRD and PRD aren't ceremony — they are the substrate AI uses to produce useful architecture, tasks, and code.
03
Continuous Validation
Quality is checked at every phase transition, not accumulated and reviewed at the end. Every phase has an entry condition and an exit gate. Gate failed = back to the current phase, not forward to the next.
04
Traceability as First-Class
BRD → PRD → Architecture → Dev Tasks → code → tests. Every step must reference the one before it. A broken chain is not a style issue — it's a gap that must be fixed before the gate passes.

THE CYCLE
Every task · Every role · Every module
The Universal Working Cycle

Evaluate → Plan → Apply → Validate.

This same four-step cycle runs through every module and every task. When something goes wrong in AI-assisted work, the diagnosis is always the same: which step was skipped?
01
Evaluate
Understand the task and its constraints before touching the AI. Read the Dev Task. Read the PRD acceptance criteria. Read the ADR. Know what you're building.
→
02
Plan
Prompt for a plan first — not output. Review it against the spec before implementation starts. Correct misalignments at plan time, not after 45 minutes of generation.
→
03
Apply
Execute. Stay grounded in the plan. Redirect the AI if it drifts. You own every line — if you can't explain what the AI produced, don't commit it.
→
04
Validate
Verify against the quality bar. Run the checklist, the tests, the gate. "Feels done" is not done. Done is when the artifact passes the bar and the gate signs off.
The cycle is introduced in Module 2 and referenced by name in every module that follows. M2: artifact production. M3: prompt refinement. M4: session management. M5: implementation. M6: building assets. M7: knowledge contribution.

WEEK 1 FRIDAY
Requirements Sign-Off · The first real handoff
Week 1 Friday — Requirements Sign-Off

This is a real handoff, not a presentation.

What gets approved today is what the Solutions Architect loads on Monday morning. A weak BRD produces a weak Architecture. A vague user story produces a vague Dev Task. The gate is real because the downstream consequence is real.
BRD passes the gate when:
  • Every requirement has a unique numeric ID (BR-01, BR-02…)
  • Requirements written in business language — no technical solutioning
  • Every stakeholder has at least one requirement addressed
  • Constraints are specific and quantified
  • Success criteria are measurable
  • Out-of-scope is explicitly listed — not implied
PRD passes the gate when:
  • Every user story references its BRD requirement
  • Every user story has at least 2 testable acceptance criteria
  • Wireframes exist for the 3 most critical flows
  • BRD ↔ PRD traceability chain is clean — no orphan requirements
The three-layer verification model
  • Self-check (Mon–Tue) — artifact owner runs the quality bar checklist before declaring ready for review
  • Peer review (Wed–Thu) — the most-affected downstream role reviews against the same checklist. SD ↔ Product. SA reviews BRD. TL reviews PRD.
  • Gate review (Friday) — cohort + facilitators, sign-off form filled, verdict recorded and committed
Reference quality bar: TaskFlow's docs/artifacts/BRD.md and PRD.md. Sign-off form: knowledge/templates/gate-signoff-template.md. Full quality bar checklists: Module 2 Trainee Guide, Part 5.

WEEK 2 MONDAY · IN-PERSON · ALL ROLES
Module 4 · Before the heaviest sessions of the training
Week 2 Monday — Context Engineering

Prompting gets you started. Context engineering keeps you going.

A great prompt produces great output in message 1. Context engineering produces great output in message 50. Week 2 is where sessions get long — full BRD + PRD loaded for architecture work, 20+ user stories in one Dev Tasks session. Without context discipline, quality degrades silently and the Friday gate fails.
Module 04 · Context Engineering & Management
  • Relevant beats abundant. Loading everything you have fills the context window with noise. Load only what the specific task needs — in dependency order, with the upstream artifact first.
  • Three session strategies. Continue when quality is high and the task is related. Compact when a section is done and you're starting the next. Start fresh when quality has degraded — this is the only real fix for a degraded session.
  • Context poisoning is a specific failure. When contradictions accumulate in the session history, corrections don't fix them. The AI tries to satisfy both conflicting instructions. The only recovery is a fresh session with clean context.
  • The main exercise: participants diagnose a real poisoned-context transcript — the same failure mode that happens in actual architecture sessions. Most participants have a "that happened to me last week" moment.
  • What this unlocks for Week 2: the Architecture Document, ADR, and Dev Tasks quality bars — the checklists used at Friday's gate.
  • Subagents as context isolation: invoke a scoped agent for a sub-task, keep the main session's context clean. Applies across Claude Code, Antigravity, and Gemini.
Source for exercise: TaskFlow/docs/training-props/poisoned-context-transcript.md. Full checklists and quality bars: Module 4 Trainee Guide, Part 6.
AI Foundational Training · Stratpoint11 / 20
WEEK 2 FRIDAY
Architecture Review + Dev Handoff
Week 2 Friday — Architecture Review + Dev Tasks Handoff

Developers pick up Dev Tasks on Monday.

Every ambiguous Dev Task is a block waiting to happen in Week 3. Every undocumented architecture decision is a surprise waiting to surface mid-implementation. The gate forces clarity now, when it's cheap to fix.
Architecture Document passes when:
  • Every PRD user story is traceable to at least one architectural component
  • Every technology choice is justified or has an ADR
  • Non-functional requirements from the PRD are addressed — not assumed
  • The deployment topology is specific enough for DevOps to start IaC
  • No BRD constraint is violated
ADRs pass when:
  • At least 2 genuine alternatives documented — not strawman options
  • Rationale references specific BRD or PRD constraints
  • Consequences are honest — positive and negative
Dev Tasks pass when:
  • Every task is 1–4 hours — larger tasks are broken down further
  • Every task has a unique ID and references its PRD user story
  • Every PRD user story is covered by at least one task
  • Dependencies are explicit — developers can self-sequence without asking SA or TL
  • No task requires a decision not yet made in Architecture or ADRs
Traceability check before Friday: every Dev Task references a PRD user story, which references a BRD requirement. Any broken link → fix it Thursday, not at the gate. Reference: TaskFlow's docs/artifacts/. Full bars: Module 4 Trainee Guide, Part 6.

Week 3 Monday — Development + Customization

Where everything lands in code. Where the AI learns your standards.

Two modules in one day. Module 5 (morning): how to implement reliably from Dev Tasks with AI. Module 6 (afternoon): how to stop re-explaining your standards every session.
Module 05 · AI-Assisted Development
Plan before implement. Read the Dev Task. Prompt for a plan — files to touch, edge cases, tests needed. Review it. Then proceed. One task per session.

Tests from acceptance criteria, not from code. If you generate tests by showing the AI the implementation, you're testing the bug not the spec. Tests must come from the Dev Task's AC list.

Safe refactoring: characterization tests first, always. The implementation loop, the debugging workflow, the self-review before every PR push — all covered.
Reference: Module_5_Trainee.docx
Module 06 · Tool Customization, Agents & Governance
Rules (AGENTS.md): always-on project guidance loaded every session — cross-tool standard (Claude Code, Cursor, Antigravity v1.20.3+). Each role adds their domain standards.

Agents: a portable markdown file with frontmatter + system prompt. Your most common recurring task, invokable by name in any tool.

Skills (SKILL.md): domain expertise loaded only when the task matches. No context bloat. Open standard since December 2025.

By Monday evening: 1 rule + 1 agent + 1 skill built, wired, and tested against a real task.

The knowledge/ structure in TaskFlow
  • knowledge/prompts/{role}/ — parameterized, tested, reusable
  • knowledge/agents/{role}/ — markdown files, portable across tools
  • knowledge/skills/{name}/SKILL.md — on-demand expertise

  • knowledge/retros/ — one document per cohort; the next facilitator's first read
