Why This Training Exists

Most teams have AI tools. Very few have a shared way to use them. The gap shows up in inconsistent quality, untraceable output, and accountability gaps when things go wrong.

Skills stay personal

Prompts, rules, and patterns are discovered individually and never shared. Teams rediscover the same things every project.

No traceability

AI output isn't connected to requirements. Nobody can trace a line of code back to a business requirement.

Accountability gaps

When AI generates something wrong, nobody owns it. No named validator, no quality gate, no audit trail.

This training changes that.

Current State

Traditional SDLC — How We Work Today

Before AI augmentation. Each role works in isolation, knowledge doesn't transfer between projects, and quality checks happen too late.

Artifact Creation
  • BRD and PRD written manually from scratch on every project
  • Architecture documents are informal or skipped under time pressure
  • Dev tasks created ad-hoc in JIRA — no consistent format or structure
  • No traceability: a line of code cannot be traced back to a business requirement
Knowledge & Skills
  • Prompts and patterns discovered individually — never shared
  • Each project starts from zero — no reuse of proven approaches
  • AI tools used ad-hoc, inconsistently, and without governance
  • When someone leaves, their knowledge leaves with them
Quality & Accountability
  • Testing deferred to the end — bugs found late are expensive to fix
  • AI output is not reviewed — nobody owns what it generates
  • No quality gates between phases — defects compound across handoffs
  • Accountability gaps when AI output is wrong

Result: Teams are busy, delivery is slow, rework is high, and AI makes it faster — but not necessarily better.

Future State

AI-Augmented SDLC — Where We're Going

AI embedded at every phase. Human accountability maintained at every gate. Every artifact traceable. Every pattern reusable.

Artifact Creation
  • AI generates BRD, PRD, Architecture, Dev Tasks — humans review and sign off
  • Every artifact references its source — full traceability from BRD to deployed code
  • Dev Tasks are structured, sized 1–4 hours, with endpoint specs and test scenarios
  • Commit messages reference task IDs — code is traceable to requirements
Knowledge & Skills
  • Shared prompt library — proven prompts saved and reused by the whole team
  • CLAUDE.md and agent specs encode project standards into the AI tools
  • Tools: Claude Code, Gemini CLI, Kiro, Figma Make — used consistently
  • MCP integrations connect AI directly to JIRA/YouTrack and Google Drive
Quality & Accountability
  • Quality gates at every phase transition — nothing moves forward unsigned
  • Every AI output has a named human validator
  • CI enforces: no merge without passing tests and task ID reference
  • Retros feed improvements back into the shared knowledge base

Result: Faster delivery, consistent quality, full traceability — and AI that improves the team instead of just the individual.

Four Non-Negotiable Principles

These govern every decision in this framework. No exceptions.

01
AI Augments, Humans Decide
Every AI output passes through human review. AI never owns decisions that carry formal accountability.
02
Structure Before Speed
Invest in artifacts before engineering begins. Teams that skip upfront structure pay for it with rework.
03
Continuous Validation
Quality gates are distributed across phases, not deferred to the end. Every phase has an entry condition and an exit gate.
04
Traceability as First-Class
Every feature must trace: BRD → PRD → Architecture → Dev Tasks → code → tests. A broken chain is a gap to fix, not ignore.

The Working Cycle

Applied at every task, every day, by every role — from BRD to deployment.

01
Evaluate
Understand the task and its constraints. Read the Dev Task, PRD AC, and Architecture ADR before touching the AI.
→
02
Plan
Ask the AI to generate a plan first — not code. Review it against the task spec before implementation starts.
→
03
Apply
Implement step by step. Read every diff. You own every line — if you can't explain what the AI wrote, don't commit it.
→
04
Validate
Test against acceptance criteria. Human reviewer checks the output. Gate passed → move on. Gate failed → back to Evaluate.
Gate failed at Validate = return to Evaluate, not skip ahead. The cycle enforces quality, not speed.

QA Test Plan + CI Pipeline

QA writes test cases from acceptance criteria before coding begins. DevOps configures the CI pipeline. TL verifies the quality gate is measurable.

Test plan coverage

  • Each known issue has a regression test that would have caught it
  • Happy path + one failure path per endpoint
  • Tool: Jest + Supertest (backend), Vitest + RTL (frontend)
CI Quality Gate (must pass before merge)
  • All tests pass — no failing tests
  • Every endpoint has at least one test

AI-Assisted Development & Delivery

Context-aware coding

Before writing a single line, load these three things into your AI tool:

  1. The relevant PRD user stories (acceptance criteria)
  2. The Architecture Document (constraints and decisions)
  3. The Dev Task you are implementing

"With this context loaded, your AI tool becomes a development partner that understands your project — not a generic code generator."

Implementing from a Dev Task
  • Read the task. Read it again. Understand the acceptance criteria.
  • Ask AI to plan before writing code: "Given this Dev Task, what is your implementation plan?"
  • Review the plan against the task spec. Does it match?
  • Implement step by step. Commit after each logical unit.
  • Every commit message references the task ID.
You own every commit. If you can't explain what the AI wrote, don't commit it.

Module 5  ·  Day 6 AM–PM  ·  2 hours  ·  All Roles

Context Engineering & Management

The context window

The AI's working memory is finite. After long sessions — or when the wrong things get loaded — quality degrades. Not randomly: there are specific warning signs.

Warning signs
  • AI forgets decisions made earlier in the session
  • AI contradicts itself within the same conversation
  • Output gets shorter, vaguer, or less specific
  • AI mixes up unrelated files or concepts
  • AI repeats the same mistake after you corrected it
  • AI references things you never mentioned
Correcting once is fine. Correcting the same thing three times = context poisoning. Stop. Compact or restart.
Compacting

Summarize session, restart with summary as context.

Clearing

Fresh start, re-load only what matters for this task.

Exercise — Diagnose and recover from a degraded context All Roles  ·  45 min
  1. Open a coding session from Module 4's exercise (or use the provided poisoned transcript at docs/training-props/poisoned-context-transcript.md)
  2. Ask Claude: "What was the key architectural decision we made about data deletion?"
  3. Evaluate the answer — is it correct? Does it match docs/artifacts/ArchitectureDocument.md?
  4. If wrong or vague: compact the session. Run /compact in Claude Code.
  5. Restart with: load Dev Task + Architecture ADR + current file
  6. Ask the same question again — compare the answer quality
Output: Context checklist for your role — what to load at the start of each session
Module 6  ·  Day 6 PM  ·  2 hours  ·  Dev + Tech Lead

Tool Customization, Agents & Governance

Rules teach the AI your standards

CLAUDE.md and .cursorrules enforce project-specific behavior.

Good rules are
  • Specific and actionable (not "write good code")
  • Tied to a real project constraint
  • Verifiable (you can check if the AI followed it)

Start with 5–10 rules. Add when you see AI make the same mistake twice. Delete rules that aren't helping.

Governance — what AI must NOT do (ever)
  • Make architectural decisions
  • Approve requirements
  • Sign off on UAT
  • Authorize deployments
Every AI output has a named human validator. This is non-negotiable.
Exercise — Extend TaskFlow's CLAUDE.md Dev + TL  ·  90 min
  1. Open .claude/CLAUDE.md — read the existing rules
  2. Identify one rule the AI has already violated in the existing codebase — how would you rewrite it to be more enforceable?
  3. Add a new rule for a constraint you discovered during codebase review (e.g., a pattern that should be consistent across all routes)
  4. Test each new rule: give Claude a task and check if it follows the rule
  5. Read knowledge/agents/README.md — find the professional agent for your next task in ultimate-ai-agents/
